Bucket Sort

labuladongOriginalAbout 2461 words

Prerequisite Knowledge

Before reading this article, you should first learn:

  • Counting Sort

Summary in One Sentence

The core idea of the bucket sort algorithm consists of three steps:

  1. Distribute the elements of the array to be sorted into several "buckets" using a mapping function.

  2. Sort the elements in each bucket.

  3. Finally, merge these sorted buckets to obtain the sorted result.

The bucket sort algorithm may not be common, but I personally find its concept very interesting because you can see traces of Merge Sort and Counting Sort in its algorithmic idea.

If you have learned all the previous algorithms in order, you will appreciate the intricate connections between these algorithms. It is fascinating to see how generations of computer experts have showcased their talents for the simple need of "sorting." As successors, why not take a moment to appreciate their ingenious ideas?

Key Points of Bucket Sort

To get straight to the point, the idea behind bucket sort is quite simple. You first distribute the elements of the array to be sorted into several buckets, sort the elements within each bucket individually, and finally merge the elements from these buckets back together in order.

Does this approach sound a bit like Merge Sort? Both divide a large array into smaller arrays for sorting, and then merge them back together. However, bucket sort is more flexible, with each of its three core steps open to variation:

  1. How do you allocate elements to be sorted into buckets? You need to determine the number of buckets and provide a mapping function.

  2. How do you sort the elements within each bucket? Theoretically, any sorting algorithm can be used, or you can mimic the Merge Sort approach by recursively running bucket sort on each bucket.

  3. How do you merge the sorted buckets? Later sections will discuss the general algorithm for merging multiple sorted lists/arrays, but that algorithm involves using a Binary Heap structure and has a complexity of O(n∗logk)O(n*logk)O(n∗logk), which is clearly not suitable here:

If we are using binary heaps, we might as well go for Heap Sort, right? Therefore, the time complexity of this merge step should not exceed O(n)O(n)O(n). To achieve this, you must design the element distribution mapping function appropriately.

Regarding these three questions, I would like to first explore the second one. Have you ever wondered, why we divide the array to be sorted into several buckets and then sort each bucket? Compared to sorting the entire array directly, is there really a difference?

The answer is, if we temporarily disregard the complexity of merging sorted buckets, sorting them separately is indeed more efficient than sorting the entire array at once.

Separate Sorting vs. Overall Sorting

Taking the simplest selection sort as an example, if I directly apply selection sort to an array of size nnn, the time complexity is O(n2)O(n^2)O(n2).

Suppose we divide the array to be sorted into kkk buckets and apply selection sort to each bucket. Will the total time complexity be greater or less than O(n2)O(n^2)O(n2)?

This is essentially a simple math problem. Assume there is a positive integer nnn, which can be decomposed into n=n1+n2+⋯+nkn = n_1 + n_2 + \cdots + n_kn=n1​+n2​+⋯+nk​. Which is greater, n2n^2n2 or n12+n22+⋯+nk2n_1^2 + n_2^2 + \cdots + n_k^2n12​+n22​+⋯+nk2​?

There are multiple ways to derive the answer. Let's consider a geometric approach:

Think of n2n^2n2 as the area of a square, while n12+n22+⋯+nk2n_1^2 + n_2^2 + \cdots + n_k^2n12​+n22​+⋯+nk2​ is the sum of the areas of several smaller squares along one side of this large square. Intuitively, these smaller squares have less area than the entire square, so clearly n2>=n12+n22+⋯+nk2n^2 >= n_1^2 + n_2^2 + \cdots + n_k^2n2>=n12​+n22​+⋯+nk2​.

From this, we understand that the total time complexity of separate sorting is less than that of overall sorting, which is the mathematical foundation ensuring the efficiency of the bucket sort algorithm.

Building on this square area abstraction, we can go further. What happens when kkk becomes infinitely large and n1,n2,⋯ ,nkn_1, n_2, \cdots, n_kn1​,n2​,⋯,nk​ become infinitely small?

The sum of the areas of the smaller squares along one side approaches zero, eventually merging with the side itself. In other words, the value of n12+n22+⋯+nk2n_1^2 + n_2^2 + \cdots + n_k^2n12​+n22​+⋯+nk2​ approaches nnn.

Thus, if bucket sort allocates elements into as many buckets as possible (maximizing kkk), with at most one element per bucket, it effectively becomes counting sort, reducing its complexity to O(n)O(n)O(n).

Even if each bucket doesn't contain only one element, as long as k>1k > 1k>1, the time complexity of bucket sort will be less than O(n2)O(n^2)O(n2). The larger kkk is, the closer the complexity gets to O(n)O(n)O(n).

Conversely, if k=1k = 1k=1, bucket sort degenerates entirely into selection sort, with a time complexity of O(n2)O(n^2)O(n2).

Of course, the above analysis does not consider the time complexity of merging sorted buckets, but as long as merging can be done in O(n)O(n)O(n) time, the overall time complexity of bucket sort remains less than O(n2)O(n^2)O(n2).
